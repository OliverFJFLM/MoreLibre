# 図書館レコメンドサービス MVP 要件定義

## 0. 背景・目的

- **課題**: 目的に合う本を探すのが大変で、候補リストを作っても近隣の図書館に蔵書がなく徒労に終わりがち。
- **目的**: 利用者が達成したい目的を入力すると、市内全館の所蔵横断で今すぐ読める本のみを推薦し、理由と順番を添えて提示、予約導線につなげる。
- **対象**: 図書館利用頻度が高い個人利用者（資格学習・研究・仕事のリサーチ・子育て・趣味など）。

## 1. スコープ

### MVP スコープ

- 市内全館の紙資料所蔵横断検索。
- 目的入力から推薦リスト（必読／準必読／発展）の生成。
- カーリル API を用いた在架・貸出・予約可確認（必須）。
- 各館 OPAC への予約導線提供。
- マイページで目的ごとの進捗（未読／読中／読了）履歴を保存。
- JIS X 8341-3:2016 AA レベルを目標としたアクセシビリティ配慮。

### 非スコープ（将来）

- 図書館カード連携による予約操作代行。
- 電子書籍 DRM 連携。
- 児童書の音声読み上げ最適化や DAISY 連携の詳細実装。
- 司書向け展示・選書支援ダッシュボードの高度化。

## 2. ペルソナ

- **ヘビーユーザー（30代・会社員）**: 短時間で最適な読書ルートを把握したい。
- **研究志向ユーザー（大学院生）**: テーマごとに体系的な読書順序と入手容易性を重視。
- **子育て層（保護者）**: 限られた時間で入手しやすい実用的な本を選びたい。

## 3. 主要ユーザーストーリー（MVP）

1. 利用者は目的を入力する（例: 「TOEIC 800 取得」「宮崎の近代史を調べたい」）。
2. システムは市内全館の蔵書を踏まえて今すぐ読める順に必読／準必読／発展リストを提示する。
3. 各冊に推薦理由（主題適合・難易度・入手容易性など）と在架／貸出／予約可のステータスを表示する。
4. 利用者は各館 OPAC の予約画面へ遷移できる。
5. マイページで目的ごとの進捗（未読／読中／読了）を記録・確認し、過去の目的も参照できる。

## 4. 体験要件（UX）

- 入口は目的入力（自由記述）に加え、期限・対象年齢・媒体など任意の補助条件を設定可能。
- 出力は必読／準必読／発展の 3 層で「今すぐ読める順」に並ぶ。
- 各アイテムに主題・レベル・所要時間目安・在架状況などの根拠表示。
- 進捗可視化はリスト単位の冊数（◯/△冊）と各本の未読／読中／読了ステータスに限定。
- 各館 OPAC の ISBN 検索・予約ページへのワンクリック導線。
- 図書館らしい落ち着いたビジュアル、派手な演出は避ける。
- キーボード操作・スクリーンリーダー対応・適切なコントラスト・フォームラベル最適化を確保。

## 5. 機能要件（Functional）

### 5.1 目的入力・解釈

- 自由記述の目的を意図分類（資格／研究／仕事／子育て／趣味／地域史など）してクエリ拡張する。
- 補助条件（期限、媒体、対象年齢など）をリランキングに反映する。

### 5.2 書誌・所蔵統合

- NDLサーチや CiNii Books など公的・学術系メタデータを検索し重複統合する。
- 主題（NDC/NDLC）、目次、要約などから内容ベースの近似で候補を抽出する。

### 5.3 在架・可否確認（必須）

- カーリル API（Check API 等）を利用し、市内全館（systemid）横断で在架／貸出／予約可を取得する。
- 結果は 5〜15 分の短期キャッシュとし、照会中は UI にステータスを表示する。
- タイムアウトや部分失敗時は段階的な結果更新と再試行を実施する。

### 5.4 推薦生成・提示

- 候補を必読／準必読／発展に層別し、「今すぐ読める順」（在架＞予約待ち短＞予約待ち長）で並べる。
- 各アイテムに推薦理由（主題適合度、難易度推定、入手容易性指標など）を表示する。

### 5.5 予約導線

- systemid ごとに OPAC 検索 URL 規則を保持し、ISBN 検索リンクを生成する。
- 予約操作は OPAC 側で行い、サービス側では個人情報を扱わない。

### 5.6 マイページ（履歴型）

- 目的の作成・一覧・詳細、各本の未読／読中／読了切替、任意の読了日入力。
- 進捗の集計（◯/△冊）と最終更新日時の表示。
- 過去の目的をアーカイブし、後から再開できる。

### 5.7 管理・運用

- 市内全館の systemid リストと OPAC URL マッピングを管理（初期は CSV 取込みでも可）。
- 推薦ロジックとデータ利用ポリシーの説明文を公開可能にする。

## 6. 非機能要件（Non-Functional）

### 6.1 性能・可用性

- 推薦生成 API の P95 レイテンシは 1.5 秒未満（在架確認は非同期更新可）。
- 月間可用性 99.9%。
- 同時アクセス 300 セッションに対応（将来拡張可能）。

### 6.2 セキュリティ・プライバシー

- 個人情報は目的と進捗のみ保持し、予約は OPAC 側で完結。
- 保存・通信の暗号化、アクセス権限分離、監査ログを整備。
- 個人情報保護法（APPI）の趣旨に沿った目的限定と保存期間管理を行う。

### 6.3 アクセシビリティ・多言語

- JIS X 8341-3:2016 AA 相当を目標とする。
- 日本語を既定とし、将来的に英語・やさしい日本語への拡張を想定。

### 6.4 観測性・運用

- 構造化ログ、エラーレポート、外形監視を実装。
- レイテンシ、在架ヒット率、予約 CVR など主要メトリクスをダッシュボードで可視化。

## 7. 連携要件

### 7.1 カーリル API（必須）

- 市内全館の systemid を取得・管理し、定期的に更新する。
- Check API で在架／貸出／予約可を取得し、ポーリングで完了を検知。
- レート・タイムアウト設計、短期キャッシュ、再試行戦略を設計する。

### 7.2 書誌メタデータ

- NDLサーチから書誌・主題・目次・書影などを取得。
- CiNii Books を学術系の補完や版・訳の正規化に活用。
- 版の束ねと ISBN（10/13 桁）の正規化を実施する。

### 7.3 OPAC 予約リンク

- systemid と OPAC 検索 URL のマッピング表を保守し、ISBN 検索クエリ差異に対応。
- 不明な場合は OPAC トップ＋タイトル検索へのフォールバックを用意する。

## 8. データ設計（概念）

- **User**: メール等の識別子（MVP では簡易認証）。
- **Goal**: タイトル、説明、期限（任意）、アーカイブフラグ。
- **Book**: ISBN、書名、著者、出版者、出版年、主題（NDC/NDLC など）。
- **GoalBook**: 目的と本の関連、推薦順、進捗ステータス（未読／読中／読了）、読了日。
- **AvailabilitySnapshot**: ISBN × systemid の在架結果（短期キャッシュ）。
- **LibrarySystem**: systemid、館名、OPAC 検索 URL 規則。
- **Retention**: 進捗・目的は継続保存、可用性スナップショットは短期保持。

## 9. 画面要件（MVP）

- **ホーム**: 目的入力フォーム（自由記述＋補助条件）。
- **結果画面**: 必読／準必読／発展の 3 タブ、各カードに書影・書誌・推薦理由・在架表示・予約ボタン。
- **マイページ（一覧）**: 目的リスト、進捗バー（◯/△冊）、最終更新日時。
- **目的詳細**: 推薦リスト、未読／読中／読了切替、読了日入力、アーカイブ操作。
- **設定**: 市内選択（デフォルト市内）、プライバシーポリシー、利用規約。

## 10. エラーハンドリング方針

- 外部 API 遅延・失敗時は在架表示を「照会中」とし、後から結果を差し替える。
- 一部館のみ取得失敗時は取得済み結果を表示し、再試行ステータスを明示する。
- ISBN 不正や版不一致時は候補から除外し、必要に応じ代替版を提示する。
- OPAC リンク不明時はタイトル検索リンクへフォールバックする。

## 11. キャッシュ・性能方針

- 在架結果は 5〜15 分の短期キャッシュ。
- systemid リストは 30 日の中期キャッシュ。
- 書誌メタデータは長期キャッシュ（年単位、更新時に差分反映）。
- 同一リクエストのデバウンスとバックオフ再試行を実装する。

## 12. セキュリティ・法務

- MVP ではメール＋パスワードの簡易認証、将来 SSO や図書館カード連携を検討。
- パスワードは強固なハッシュ化を行い、セッション管理は JWT など標準手法を利用。
- 個人情報の取り扱い範囲を「目的・進捗・メール」に限定し、予約は OPAC 側で完結。
- 利用規約・プライバシーポリシーを整備し、データの目的外利用を禁止。

## 13. 受け入れ基準（Definition of Done）

- 目的入力→推薦→在架確認→OPAC 予約リンクまでの一連の操作が可能。
- マイページに目的が保存され、進捗切替で進捗バーが更新される。
- 推薦生成の P95 レイテンシが 1.5 秒未満（在架は非同期更新可）。
- アクセシビリティ簡易試験（キーボード操作・読み上げ・コントラスト）を通過。
- ログ・メトリクス・外形監視が稼働。
- 運用 Runbook（障害対応、レート制御、キャッシュクリア手順）が整備されている。

## 14. KPI（MVP）

- 目的入力から予約リンククリックまでのコンバージョン率。
- 在架ヒット率（在架で提示できた割合）。
- 予約待ち日数の中央値（取得可能範囲で推定）。
- 再訪率（週次）と目的完読率（自己申告）。

## 15. リスクと対策

- **外部 API 依存**: キャッシュ、フォールバック、再試行、タイムアウト閾値を明確化。
- **OPAC 仕様差**: systemid ごとの URL ルールをマスタ管理し、未知館はフォールバック。
- **推薦の偏り**: 年代・難易度・出版年の分布を監視し、同質化を防ぐ。
- **版・訳の混在**: ISBN 正規化と版の束ねで対応。

## 16. 納品物（MVP 完了時）

- 画面仕様（ワイヤーフレーム、UI 指針、アクセシビリティ方針）。
- 機能仕様（本書＋更新差分、外部 API I/F 定義、在架取得の状態遷移図）。
- データ設計（概念 ER、主要プロパティ一覧、保持期間・削除方針）。
- 運用設計（監視項目、しきい値、障害時フロー、キャッシュ戦略）。
- テスト仕様（正常系・異常系・遅延時の受入観点）。
- 導入手順書（環境変数、連携キー設定、systemid/OPAC マッピング投入手順）。
- リリースノート（既知の制約、次期開発論点）。

## 17. ローンチ計画（例）

- **W1–2**: systemid/OPAC マッピング収集、外部 API 鍵取得、画面ワイヤー確定。
- **W3–4**: 目的入力→推薦→在架連携→予約導線の縦串実装。
- **W5–6**: マイページ（進捗履歴）、アクセシビリティ対応、受入テスト。
- **W7**: パイロット（ヘビーユーザー 30〜50 名）とフィードバック反映。
- **W8**: 本番公開、KPI 監視開始。

## 18. 成功の定義（MVP）

- 「水の泡リスト」が解消され、今すぐ借りられる順で読書が始められる。
- 進捗が着実に蓄積され、再訪・継続利用につながる。
- 図書館らしい落ち着いた体験と説明可能性を両立する。
